### admin-server-1. [MED] cron.ts docstring falsely claims validateEnv hard-fails on CRON_SECRET

`apps/admin/src/lib/server/cron.ts:16` — inconsistency, confidence: certain

The docstring (lines 16-17) states 'Fail-closed: an unset CRON_SECRET rejects every request (validateEnv already hard-fails on it in prod).' But apps/admin/src/lib/server/validateEnv.ts REQUIRED = ['DATABASE_URL','BETTER_AUTH_SECRET','ORIGIN'] does NOT include CRON_SECRET, and nothing else validates it. Consequence: if CRON_SECRET is unset in prod there is no boot-time failure or warning; requireCron silently returns 401 on every call, so /api/network/health/refresh stops warming network_health with no visible signal. Either add CRON_SECRET to validateEnv (matching the claim) or fix the docstring.

> **Fable verification (high effort): `CONFIRMED`** — cron.ts:16-17 claims 'validateEnv already hard-fails on it in prod', but validateEnv.ts:19 REQUIRED = ['DATABASE_URL','BETTER_AUTH_SECRET','ORIGIN'] and no other code validates CRON_SECRET (grep confirms only cron.ts/rateLimit.ts comments reference it). requireCron at cron.ts:26 does fail closed (401 when unset), so the consequence is exactly as stated: silent 401s on /api/network/health/refresh with no boot signal.
